# AI3D вҖ” Interactive 3D Model Generator

**Date:** 2026-03-16
**Status:** Draft

## Overview

AI3D is an interactive Rust CLI that uses Claude as a CAD engineer to generate functional 3D models from natural language descriptions. The user describes a part, Claude generates CadQuery code, a Python subprocess builds the STL, and the Rust TUI handles results display and iterative refinement. Designed for resin printing workflows where STL export is the endpoint.

## Goals

- Generate functional/mechanical 3D parts (brackets, enclosures, mounts) with dimensional accuracy
- Rapid prototyping вҖ” quick physical mockups to test form and fit
- Interactive refinement loop: describe вҶ’ generate вҶ’ review вҶ’ refine вҶ’ export
- No chat gateway or bot framework вҖ” direct CLI tool with Claude API integration

## Non-Goals

- Slicing and print control (user handles this in their resin slicer)
- AI image-to-3D pipelines (Tripo, Hunyuan) вҖ” these produce organic meshes unsuitable for functional parts
- Web UI or chat interface
- FDM-specific features (bridging, support optimization)

### Component Responsibilities

**Rust binary (`ai3d`):**
- Interactive session loop (ratatui + crossterm)
- Claude API client (reqwest, streaming responses)
- Conversation history management
- Terminal 3D preview (braille character projection)
- External viewer launching (f3d)
- STL binary reader (for preview rendering)
- Configuration loading

**Python package (`ai3d-cad`):**
- CadQuery code execution and STL export
- OpenSCAD fallback (system binary, headless mode)
- Mesh analysis (dimensions, volume, triangle count, watertight check)
- Code validation (syntax check without building)
- No AI logic, no UI, no network calls

## Interactive Session Flow

### Session States

1. **Idle** вҖ” waiting for user input (new design or refinement)
2. **Clarifying** вҖ” Claude asks questions before generating
3. **Generating** вҖ” Claude produces code, Python builds STL
4. **Reviewing** вҖ” model built, user sees summary, can preview/refine/export
5. **Error** вҖ” build failed, Claude sees the error and suggests fixes

### User Commands

| Command | Action |
|---------|--------|
| (free text) | Describe what to build, or refine current model (prefix with `r` as shorthand, e.g. `r add fillets`) |
| `show` / `s` | Open current STL in external viewer (f3d) |
| `preview` / `p` | Terminal braille wireframe preview |
| `export <file>` / `e` | Save STL to specified path |
| `code` | Show the current CadQuery/OpenSCAD source |
| `history` | Show design iteration history |
| `new` | Start a fresh design (resets conversation) |
| `undo` | Revert to previous iteration (single level; restores code, STL, metadata, and conversation state) |
| `help` | List commands |
| `quit` / `q` | Exit |

## Claude API Integration

### System Prompt

```
You are a CAD engineer assistant. You generate CadQuery Python code
that produces 3D models for resin 3D printing.

Rules:
- Output ONLY a ```cadquery code block + brief explanation
- All dimensions in millimeters
- Design for resin printing (no FDM-specific features like bridging)
- Prefer CadQuery. Fall back to OpenSCAD only if the user requests it
  or the geometry is better expressed as CSG
- When refining, modify the existing code вҖ” don't rewrite from scratch
- If the user's request is ambiguous, ask ONE clarifying question
  instead of guessing
- Annotate features with # feature: comments in the code
```

### Conversation Context

The Rust client maintains message history and sends it with each API call:

- System prompt is constant
- Full conversation history for the current design
- On `new` command, history resets
- Model metadata (dimensions, features) is injected as context before each request
- **Context management:** When conversation exceeds 20 exchanges, older exchanges are summarized into a single context message retaining key decisions and the current working code. The most recent code + metadata always remain in full. Undo snapshots capture the conversation state at each iteration's completion (post-summarization if applicable), so undo always restores a consistent state.

### Response Parsing

Claude's response is parsed for:
1. A ` ```cadquery ` code block вҖ” sent to Python with `--engine cadquery`
2. A ` ```openscad ` code block вҖ” sent to Python with `--engine openscad`
3. A ` ```python ` code block вҖ” treated as CadQuery only if it contains `import cadquery`; otherwise displayed as illustrative code
4. Plain text (clarifying question or explanation) вҖ” displayed to user
5. Both code and text вҖ” code is built, text shown alongside results

### Error Recovery

When Python fails to build, the error is fed back to Claude with the original code and user intent. Max 3 auto-retries before surfacing the error to the user.

### Streaming

Claude API is called with streaming enabled. Text tokens are displayed in the TUI as they arrive (live "thinking" feedback). Code block extraction and build triggering happen only after the full response is received, since partial code blocks are ambiguous.

### Model Selection

Default: Claude Sonnet for speed (2-5s generation). Configurable via config file or env var.

## Subprocess Protocol

The Rust binary communicates with the Python package via subprocess calls with a well-defined contract.

### Invocation

The Rust side writes Claude's generated code to a temp file in the session directory, then invokes:

```bash
python -m ai3d_cad build --code /tmp/ai3d-XXXX/iter_003.py --output /tmp/ai3d-XXXX/iter_003.stl --engine cadquery
python -m ai3d_cad build --code /tmp/ai3d-XXXX/iter_003.scad --output /tmp/ai3d-XXXX/iter_003.stl --engine openscad
```

The `--engine` flag explicitly selects the backend. Rust determines the engine from Claude's response (` ```cadquery ` вҶ’ cadquery, ` ```openscad ` вҶ’ openscad).

### Exit Codes

| Code | Meaning | stdout | stderr |
|------|---------|--------|--------|
| 0 | Success | JSON metadata | Warnings (if any) |
| 1 | Build error (bad geometry, CadQuery exception) | `{"error": "...", "error_type": "build"}` | Full stack trace |
| 2 | Validation error (syntax) | `{"error": "...", "error_type": "syntax"}` | Details |

### Timeout

Default build timeout: 60 seconds (configurable via `build_timeout` in config). The Rust side sends SIGTERM, waits 5 seconds, then SIGKILL if the process hasn't exited. CadQuery with Open CASCADE can hang on degenerate geometry (e.g., impossible fillets, boolean ops on zero-thickness shells). On timeout, Rust treats the result as a build error internally regardless of the process exit code, and reports "Build timed out after Ns" to the error recovery loop.

### Temp File Lifecycle

Each session creates a temp directory (`/tmp/ai3d-XXXX/`). All intermediate code files, STLs, and metadata live there. Structure:

```
/tmp/ai3d-a1b2c3/
в”ңв”Җв”Җ iter_001.py          # CadQuery code, iteration 1
в”ңв”Җв”Җ iter_001.stl         # Built STL
в”ңв”Җв”Җ iter_001.json        # Metadata
в”ңв”Җв”Җ iter_002.py          # Iteration 2 (refinement)
в”ңв”Җв”Җ iter_002.stl
в”ңв”Җв”Җ iter_002.json
в””в”Җв”Җ ...
```

Additionally, a `current.stl` symlink always points to the latest iteration's STL (used by f3d `--watch` for auto-reload).

Cleaned up on `quit`. On `new`, the directory is wiped and iteration counter resets. On `export`, the STL is copied to the user-specified path.

## Python Package (`ai3d-cad`)

### CLI Interface

```bash
# Build: execute CadQuery/OpenSCAD code, produce STL + metadata JSON sidecar
python -m ai3d_cad build --code script.py --output model.stl --engine cadquery
# Writes: model.stl + model.stl.json (metadata sidecar)
# stdout: JSON metadata (exit 0) or JSON error (exit 1/2)

# Info: analyze an existing STL
python -m ai3d_cad info --input model.stl
# stdout: JSON metadata

# Validate: syntax-check code without building
python -m ai3d_cad validate --code script.py --engine cadquery
# stdout: {"valid": true} or {"valid": false, "error": "..."}

# Version: protocol version for compatibility check
python -m ai3d_cad --version
# stdout: ai3d-cad 0.1.0 (protocol 1)
```

### Metadata JSON Output

```json
{
  "dimensions": {"x": 92.0, "y": 62.0, "z": 6.5},
  "volume_mm3": 14280.5,
  "triangle_count": 3420,
  "features": ["4x through-hole M2.5", "4x standoff 4mm", "fillet 3mm"],
  "watertight": true,
  "engine": "cadquery"
}
```

The `features` list is best-effort, extracted from `# feature: ...` comments in CadQuery code. May be empty if Claude omits annotations.

### Dependencies

- `cadquery` вҖ” core CAD engine (BREP kernel via Open CASCADE)
- `trimesh` вҖ” STL export, mesh analysis, watertight check
- `numpy` вҖ” geometry operations (transitive via cadquery)
- OpenSCAD support uses the system-installed `openscad` binary (no Python deps)

### Installation

CadQuery requires conda/mamba for its Open CASCADE bindings. Recommended setup:

```bash
mamba create -n ai3d python=3.11
mamba activate ai3d
mamba install -c cadquery cadquery
pip install ai3d-cad    # or: cd python/ && pip install -e .
```

The Python package ships an `environment.yml` for reproducible setup.

## Terminal Preview

Braille character rendering of STL meshes directly in the terminal:

- Reads binary STL (triangle list)
- Orthographic projection onto terminal grid
- Braille dot characters for ~2x horizontal, 4x vertical sub-character resolution
- Arrow keys to rotate in 90В° steps between standard views
- Sized to fit terminal width
- No GPU required

Purpose: quick "does this look roughly right?" check without leaving the terminal.

## External Viewer

Launch `f3d` (or configurable alternative) as a detached process, pointing at the `current.stl` symlink with `--watch` so it auto-reloads on each iteration:

```bash
f3d --watch /tmp/ai3d-XXXX/current.stl &
```

The `show` command launches f3d once per session. Subsequent iterations update the symlink, and f3d picks up the change automatically. If f3d is already running, `show` is a no-op.

Fallback chain: `f3d` вҶ’ `meshlab` вҶ’ `xdg-open` вҶ’ error with install instructions. Fallbacks other than f3d don't support `--watch` and are re-launched on each `show`.

## Configuration

File: `~/.config/ai3d/config.toml`

```toml
[claude]
api_key = "sk-ant-..."       # or ANTHROPIC_API_KEY env var
model = "claude-sonnet-4-6"  # default model

[viewer]
command = "f3d"               # external viewer binary

[defaults]
output_dir = "."              # where STLs are exported
max_retries = 3               # auto-retry on build failure
build_timeout = 60            # seconds before killing Python subprocess
```

Environment variables override config file values. `ANTHROPIC_API_KEY` is the only required setup.

## Startup Checks

On launch, the Rust binary verifies:
1. `ANTHROPIC_API_KEY` is set (env var or config file)
2. `python -m ai3d_cad --version` succeeds and reports a compatible protocol version (Rust requires `protocol 1`; mismatch is fatal with upgrade instructions)
3. Optionally checks for `f3d` and reports if missing

Clear error messages with setup instructions if any check fails.
